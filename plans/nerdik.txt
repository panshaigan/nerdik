nerdik (codename)

A system for organizing and participating nerd events like RPG sessions, board game meetings, lectures, conventions.
The first release focuses on in-person nerd events and crucial features.

Initial implementation order for ver 0.1
    - Dev environment & scaffolding
        - I used `curl -s "https://laravel.build/nerdik?with=mariadb,mailpit,redis" | bash` to create the env. its already working
        - Configure .env if needed
        - please execute every needed command through sail if needed
    - Core auth & users
        - Breeze (email+password + Google).
        - Extend users table with nickname, avatar, discord, languages, location etc.
    - Places & tags (foundations)
        - Migrations + basic CRUD UIs.
        - Seed Polish cities and core tags (popular RPGs + triggers).
    - Events & event_instances
        - Models, migrations, basic user CRUD.
        - Attach organization (simple CRUD).
    - Slots
        - Slot creation with mass generation (N tables at time T, type X, place Y).
    - Activities & proposals
        - Activity CRUD (just enough fields).
        - Proposal flow:
            - Propose to event_instance and/or slots.
            - Auto‑accept on “no approval needed” slots.
    - Participation & waitlists
        - Join/leave logic (with waitlist).
        - Simple email notifications hooked into queue.
    - Basic browsing & dashboard
        - Event and activity listings (filters: date, place, tags).
        - Simple user dashboard lists.

App Premises:
 	- web app
	- keep low costs (e.g. minimum uploads - just avatars and logos)
	- free for free activities which should strengthen the fandoms
	- connecting small group of nerds with similar interests
	- helps to organize signing up for activities during bigger events
	- mobile friendly from the beginning + possibility to add mobile app in the future
	- i want to learn new stuff. laravel, frontend, maybe kubernetes.
	- notifications should be both in-app and by external methods, like email for the beginning. mobile/sms in the future

User Roles:
	- event organizer
	- activity host (e.g. GM)
	- regular participant
	- they are no special kind of users. they are determined just by the creation of entities. if you create an event, you are the event organizer. if you create an activity, you may add yourself as the activity host if the activity type needs one (e.g. RPG sessions, lectures)
	- the only special kind of user are Admins, who have godmode on everything

UX & Workflows {TO CHECK}:
	Key flows:
		Creating an event and its instances
			- Create event
			- Create one or more event_instances by choosing the dates/date ranges in calendar. the name/desc/places/tags are initially copied from the main event
			- Adjust event_instances on need
		Creating slots within instance
			- Open event instance. 
			- Choose base name (e.g. "Table"), activity type(s), place, time (within instance's time range), need approval flag
			- Set count (e.g. 10 slots for RPG at 13:00 or 1 slot for board games)
			- Generate N slots in one action.
		Proposing an activity flow
			- User goes to an event and clicks either "propose an activity" either clicks a conrete slot if available
			- User creates activity (or picks existing from the ones he created before; duplicate if already assigned somewhere). 
				- if he clicked a concrete slot, his new activity form is prefilled with duration, activity type (if there was only one possible) according to the slot
				- if he clicked general "propose an activity", he may mark slots which are good for him or chooses preferable start time or leave it free to decide by the organizer
			- Clicks "Propose"
			- If there is any "no approval needed" slot, proposal is auto-accepted and activity is attached to one such slot. 
			- Otherwise proposal waits; 
				- organizer accepts - activity gets attached to the chosen slot, or if there was no chosen slot, organizer can see the prefered slots and choose it himself. can be by drag and drop
				- reject - activity stays in user's list for reuse/stand-alone.
			- technicals:
				- User creates a proposal and attaches it to an event instance and optionally a set of compatible slots (same type) via activity_proposal_slot or just set the preferred time or leaves everything for the organizer's decision.
				- If any of those slots has "no approval needed", auto-accept: set accepted_slot_id and set status = accepted, set slot.activity_id = activity_id.
				- Otherwise organizer chooses one slot from that set and set the same properties
		Joining/leaving an accepted activity
			- User joins an accepted activity (if capacity allows). 
			- If full or only-with-host-confirmation, optional "join waitlist"
			- On leave (if time allows), optional refill from waitlist. (the new participant should be notified)
			- Capacity and "slots for users" come from activity.
		Browsing activities
		 	- From event instance: list accepted activities (and maybe free slots). 
		 	- From dashboard: my participations, my proposals, wishlist, owned activities/events
		 	- from events/activities catalog
	 	Tag system
			- One concept with:
				- Multiple labels in different languages. [MVP]
				- Hierarchy (Fri Ligan > Forbidden Lands). [MVP]
				- Aliases [FUTURE]
                - Attached tags [FUTURE]
			- Search that:
				- Matches both "Zakazane Ziemie" and "Forbidden Lands". "WFRP" and "Warhammer Fantasy Roleplay"
				- Treats them as the same base tag.
			- Behavior:
				- When user adds a tag:
					- He types a label; you search tag_translation.label and tag_alias.alias (case-insensitive, maybe fuzzy).
					- If match found, link to that tag_id.
					- If not:
						- Create new tag and tag_translation for the user’s locale, and
						- Queue for admin review to eventually merge with existing tags.
				- When user searches:
					- Search both translations and aliases in their locale and also locale‑agnostic synonyms.
